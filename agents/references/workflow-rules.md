# 工作流规则

本文件供 SKILL.md 全局引用。包含贯穿所有阶段的行为准则、输入处理和流程控制规则。

## 批判框架

当产品想法价值弱或逻辑不成立时，按固定结构输出：

1. **结论**：当前方向是否成立。
2. **致命问题**：用户、场景、频率、痛感、替代方案、验证方式中哪一项不成立。
3. **为什么严重**：如果不改，后续 PRD、原型或开发会如何浪费。
4. **修正选项**：收窄场景 / 换目标用户 / 降级为验证实验 / 暂停方向。
5. **推荐动作**：明确推荐一个下一步。
6. **需要用户拍板**：继续、收窄、换方向或放弃。

## 输入要求

用户需提供以下信息（缺失项在对应阶段主动追问）：

| 字段 | 必填 | 说明 |
|------|------|------|
| 功能描述 | 是 | 要做什么功能、解决什么问题，至少一句话 |
| 目标用户 | 否 | 面向谁，未提供则 S2 追问 |
| 业务目标 | 否 | 期望达成的业务指标（DAU、转化率、营收等），未提供则 S2 追问，目标不含数字时标注 `[待量化]` |
| 产品类型 | 否 | B2C/B2B/内部工具/平台型/混合型，未指定则 S1 推断 |
| 约束条件 | 否 | 技术限制、时间要求、资源限制，未提供则 S1.5 追问 |
| PRD 深度 | 否 | 概要版（评审用）/ 详细版（开发用），默认详细版 |

## 信息完整度与模式分流

- **快速模式**：用户提供了功能描述 + 目标用户 + 业务目标时进入。减少追问轮次，加速进入正式产物阶段；但仍需完成 S1 类型分流和 MVP 收口门禁。
- **引导模式**：用户仅说"写个 PRD"未给需求时进入。追问功能描述，给出示例引导："比如：做一个社区积分商城，面向 20-35 岁女性用户，目标是提升社区留存率 20%。"
- **概要模式**：只有一句话需求（如"做个商城"）时进入。生成概要版 PRD 框架，标注需要补充的关键信息，不强行填完所有模板字段。
- **膨胀警告**：需求过于庞大时，建议拆分为多个 PRD，先确定 MVP 范围。

## 追问规则

正常澄清阶段：
- 一次只问一个当前最关键问题。
- 每个问题必须给推荐答案。
- 用户回答后沿该分支继续深挖，直到闭合再回主线。
- 回答模糊时继续追问，不把模糊回答当确认。

快速产出或门禁前：
- 一次列出所有阻塞项。
- 每项给推荐答案。
- 询问用户是否全部接受推荐答案，或逐项修改。
- 用户确认前不得生成正式产物。

连续两次模糊回答：
- 切换为缺口清单模式。
- 明确指出哪些回答不可执行。
- 给出推荐选项让用户拍板。

## 阶段闭合输出模板

每个阶段闭合时都必须输出固定结构，不能只凭感觉切换阶段：

~~~markdown
## 阶段闭合：Sx <阶段名>

### 已确认
- 最多 5 条关键判断

### 未确认
- 无 / 列出仍会阻塞下一阶段的问题

### 推荐答案
- 对未确认项给出推荐答案；没有未确认项则写"无"

### 下一步
- 进入 Sy / 停在当前阶段继续追问
~~~

进入下一阶段的条件：
- `未确认` 为"无"；或
- 未确认项不影响下一阶段判断，且已明确标记为未验证假设、风险或后续验证项。

阶段闭合后覆盖更新 `pm-memory/PROJECT-STATE.md`。不得把模糊回答当作已确认判断。

## S3 联网与调研规则

联网规则：
- 涉及市场、竞品、行业事实、用户数据或外部趋势时，必须询问用户是否联网验证。
- 用户同意后再联网；若当前环境无联网能力，提醒用户选择跳过或使用 `web-access` 等替代方案。
- 用户拒绝或无法联网时，继续推进，但将相关判断标为未联网验证假设。
- 到 S7 正式产物前，如果仍有外部事实依赖，再次提醒是否补调研。

调研材料：
- 调研结果作为材料进入 `pm-output/iterations/<version>/调研材料.md`。
- `pm-memory` 不保存调研资料，只保存阶段和版本状态。
