# 环境识别与约束分流

当 PM Assistant 进入 `S1.5`，或即将生成 `PRD.md` / `modules/` 时读取本文件。

目标：
- 判断产品所处环境，决定主 PRD 和模块详设需要下钻到什么颗粒度。
- 识别已有系统、跨端、授权、计费、审核、算法、合规、发布验证等会影响文档结构的约束。
- 建立“主 PRD 写全局环境，模块详设写继承 / 覆盖”的规则。

## 必须问清

| 环境类型 | 必须确认的问题 |
|----------|----------------|
| 产品环境 | 新产品、已有产品迭代、增量功能、重构、下线还是迁移？ |
| 端环境 | App、Web、H5、小程序、后台、插件、多端联动？是否涉及 H5 / App 呼起？ |
| 现有系统环境 | 已有页面、已有数据、线上逻辑、历史用户、老版本兼容是什么？ |
| 业务环境 | 服务增长、留存、转化、运营、商业化、风控还是成本目标？ |
| 技术依赖 | 是否涉及登录、权限、接口、算法 / 模型、异步任务、消息、支付、积分、第三方服务？ |
| 合规 / 安全 | 是否涉及隐私、授权、内容安全、支付、未成年人、声音 / 图片 / 人脸等敏感数据？ |
| 发布验证 | 是否需要灰度、AB、开关、回滚、埋点、告警、成功 / 异常判断？ |

## 环境分流规则

| 命中环境 | 必须下沉到模块详设的内容 |
|----------|--------------------------|
| 已有产品迭代 | 现有逻辑与本期改动 |
| 多入口 / 多端 | 入口矩阵、跳转与回流规则 |
| 邀请 / 分享 / 授权 / 协作 | 角色 / 主客态链路、权限、隐私、风控与安全 |
| 支付 / 积分 / 商业化 | 消耗规则、失败回滚、权益影响、对账 / 风控 |
| 审核 / 算法 / 生成任务 | 异步状态机、失败态、重试、人工或系统兜底 |
| 旧数据 / 老版本 | 兼容风险、迁移规则、灰度 / 回滚方案 |
