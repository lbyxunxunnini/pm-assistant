# 正式输出规则

当 PM Assistant 进入 `S7`，或需要写入 `pm-output/iterations/<version>/` 时读取本文件。

## 版本规则

- 创建正式产物前必须确认版本号。
- 如果是首个正式版本，推荐 `1.0.0`，但必须由用户确认。
- 如果已有版本，询问是创建新迭代还是覆盖 / 修正当前版本。
- 默认禁止覆盖旧版本，优先创建新迭代。
- 用户定不下版本变化时，提供：
  - `+0.0.1`：小修订，文案、规则、状态、验收补充。
  - `+0.1.0`：中迭代，新增 / 调整功能或主流程，产品方向不变。
  - `+1.0.0`：大版本，目标用户、核心场景、产品方向或商业目标明显变化。

## 写入前版本检查

- 每次写入正式产物前，必须先扫描 `pm-output/iterations/` 下已有版本目录；若 `pm-memory/OUTPUTS.md` 存在，也必须读取其中的版本索引。
- 不得只依赖 `pm-memory/PROJECT-STATE.md` 判断是否为首个版本；checkpoint 可能缺失、过期或只记录“未建版”。
- 将用户确认的目标版本与现有版本目录和 `OUTPUTS.md` 记录比对。
- 若目标版本已存在，必须暂停并让用户选择：创建新版本、修正当前版本，或明确覆盖。
- 默认禁止覆盖已有版本目录。只有用户明确说“覆盖 / 修正当前版本 / 写入这个已有版本”时，才允许写入同名版本。
- 如果 `PROJECT-STATE.md`、`OUTPUTS.md` 和版本目录互相矛盾，先输出冲突摘要和推荐处理方式，不得直接写入。

## 输出目录

```text
pm-output/
└── iterations/
    └── <version>/
        ├── PRD.md
        ├── modules/
        │   ├── 01-<模块名>.md
        │   └── 02-<模块名>.md
        ├── 开发交接.md
        ├── 迭代摘要.md
        ├── 决策日志.md
        ├── 未决问题.md
        └── 调研材料.md
```

## 默认收敛规则

- 默认正式输出生成 `PRD.md`。
- 如果产品包含多个功能模块、已有产品迭代、多入口链路、邀请 / 分享 / 授权 / 支付 / 积分 / 审核 / H5-App 跳转等复杂场景，必须额外生成 `modules/` 下的模块详设文件。
- 如用户要求开发交接，再额外生成 `开发交接.md`。
- 不生成独立产品决策包、交互说明、原型 Brief 或验收标准文件。产品决策、交互、原型布局和验收全部进入 `PRD.md` 与 `modules/`。
- PM Assistant 不生成 `design-output/`。具体视觉稿、HTML/CSS/SVG 原型、截图验证和可视化交付属于设计 / 原型工具职责。

## PRD 完整性门禁

- `PRD.md` 必须包含：产品背景、业务目标与成功指标（SMART 格式）、目标用户、痛点、替代方案、价值假设、证据状态、全局环境与约束、技术选型或技术假设、MVP 范围、非目标、功能模块总览（含用户故事列和 P0/P1/P2 优先级列）、模块详设索引、总体流程图、信息架构 / 页面地图、核心页面原型布局、跨模块交互、全局状态与边界、全局数据 / 权限、全局数据需求（埋点事件表）、非功能需求（表格格式含验收标准列）、验收标准、排期建议、风险与未决项。
- 总体流程图、信息架构 / 页面地图必须使用 Mermaid；核心页面原型布局必须使用可读的文本线框图或 Mermaid，不得只写页面清单。
- 功能模块总览的用户故事列必须使用"作为{角色}，我希望{操作}，以便{价值}"格式。
- 业务目标与成功指标必须 SMART 化——有具体数字和截止时间；用户未提供数字时标注 `[待量化]`。
- 全局数据需求节必须列出核心操作路径的埋点事件，与模块详设的埋点节建立继承 / 覆盖关系。
- 排期建议必须拆到开发 / 测试 / 联调 / 上线四个阶段；用户未提供工时时标注 `[待补充]`。
- 业务规则不允许出现"等""其他情况""按业务规则处理"等模糊表述；无法穷举时必须标记为未决项。
- 每个核心功能模块至少覆盖 2 个异常场景。
- `PRD.md` 不得承载所有模块细节导致不可读。真实线上颗粒度必须进入 `modules/<序号>-<模块名>.md`，并在主 PRD 中用模块索引链接。
- 主 PRD 的全局环境必须明确哪些环境项需要模块继承，哪些环境项允许模块覆盖。
- 技术选型不是 PM Assistant 可擅自决定的内容。若用户没有确认，必须写成“推荐 / 假设 / 待确认”，并说明该选型对 PRD 的影响。

## 模块详设门禁

- 每个模块详设文件必须达到真实产品迭代 PRD 的颗粒度，至少包含：模块目标、适用范围、模块环境继承与覆盖、现有逻辑与本期改动、入口矩阵、角色 / 主客态链路、页面与弹层线框图、详细交互说明、业务对象状态机、跳转与回流规则、文案资产表、数据与接口字段、异常 / 并发 / 权限 / 风控规则、埋点指标、验收用例。
- 已有产品迭代必须写清“原逻辑 / 本期新增 / 本期修改 / 本期删除 / 复用线上逻辑 / 兼容风险”。
- 邀请、分享、授权、协作类模块必须拆清发起方、接收方、被影响方、第三方访问者、多设备 / 未登录 / 未安装状态。
- H5 / App 呼起 / 跨端链路必须写清安装状态、登录状态、版本升级、弹窗优先级、失败降级、完成后回流页面。
- 计费、积分、审核、合成任务等模块必须写清消耗规则、失败回滚、异步状态、风控提示和用户可恢复路径。

## 正式输出完成门禁

- 进入正式写入前必须确认 S1.5 已闭合；若本版本需要 `modules/`，必须确认 S5.6 已闭合。
- 每次生成或更新正式产物后，不得直接宣称完成；必须立即进入 S8，执行与本次产物相关的一致性审查。
- 如果只生成单个正式产物，也必须至少核对该产物与 `PROJECT-STATE.md`、MVP 范围、未验证假设和验收标准是否一致。
- 如果生成 `modules/` 或其他多个正式产物，必须使用 S8 的完整一致性清单检查主 PRD 与每个模块详设之间是否冲突。
- 只有 S8 审查通过、必要的 `迭代摘要.md` / `PROGRESS.md` / `OUTPUTS.md` 已更新后，才能宣称本次正式输出完成。
- 若 S8 发现冲突，必须列出冲突、影响和推荐修正；修正或用户明确接受风险前，不得宣称完成。

## 原型边界

- PM Assistant 只在 `PRD.md` 和 `modules/` 中提供页面线框、信息层级、交互状态、视觉约束和验收要求。
- PM Assistant 不生成独立 `原型Brief.md`、`UI-Forge交接.md`、HTML/CSS/SVG 原型或 `design-output/`。
- 用户要继续做具体视觉稿、可交互原型或截图验证时，应把 `PRD.md` 和 `modules/` 交给专门的 UI / 原型工具处理。

## 默认交付规则

默认完整运行的对话终点是**产品决策摘要**，默认保存终点是**完整 `PRD.md`**，不是自动生成一堆拆分文件。

### 默认闭合内容

- 产品名与 checkpoint
- 产品类型分流（B2C / B2B / 内部工具 / 平台型 / 混合型）
- 目标用户与核心场景
- 核心问题、价值假设、替代方案
- MVP 范围与非目标
- 关键规则、状态、边界
- 产品维度、视觉交互原则和核心组件 / 按钮行为
- 验收标准与成功信号
- 当前阶段进度与下一步建议

### S6 默认收尾规则

- S1-S6 全部闭合且 S1.5 已闭合后，必须先在对话中输出产品决策摘要；若本轮涉及页面线框、视觉约束、模块详设或开发交接，S5.5 也必须已闭合；若符合模块详设触发条件，S5.6 也必须已闭合。
- 随后询问用户是否保存为完整 `PRD.md`。该 PRD 必须整合产品决策、需求、流程、交互、原型布局、验收和风险。
- 用户确认保存时，进入 S7，确认版本号，并默认写入 `pm-output/iterations/<version>/PRD.md`；符合模块详设触发条件时，同时写入 `pm-output/iterations/<version>/modules/`。
- 用户拒绝保存时，必须明确提示：当前只保留 `pm-memory/PROJECT-STATE.md` checkpoint，完整 PRD 不会被持久化，后续恢复只能依赖 checkpoint 和当前对话上下文。
- 用户未表态时，不得宣称正式产物已完成。
