# PRD 质量自检清单

当 PM Assistant 完成正式产物生成后、进入 S8 一致性审查前，必须对本清单逐项校验。

本清单只校验文档自身完整度，不替代 S8 的跨产物一致性审查。

## 自检清单

| 序号 | 检查项 | 判断标准 | 不通过时的处理 |
|------|--------|----------|------------------|
| 1 | 背景不空泛 | 回答了"为什么做"而非只说"需要做"；背景描述含用户痛点、市场时机或业务压力，而非泛泛的"业务需要" | 标注`[背景待补充]`并提示用户补充具体痛点或时机 |
| 2 | 目标可量化 | 至少一个目标含具体数字（如 DAU、转化率、营收、日活等）；若目标无法量化，标注`[目标待量化]`并给出可量化的推荐指标 | 不通过时标注`[目标待量化]`，建议用户补充数字型成功指标 |
| 3 | 用户画像具体 | 不含"所有用户"这种泛描述；每个用户角色有特征描述、场景和诉求 | 标注`[用户画像待细化]`，建议补充角色-特征-场景-诉求格式 |
| 4 | 业务规则穷举 | 无"等"/"其他情况"/"按业务规则处理"等模糊表述；每条规则有明确触发条件、处理逻辑和边界值 | 标注`[规则黑洞]`，要求逐条补齐触发条件、处理逻辑和边界值 |
| 5 | 异常流程覆盖 | 每个功能模块至少覆盖 2 个异常场景（网络超时、数据异常、权限不足等） | 标注`[异常覆盖不足]`，列出缺少的异常场景并建议补充 |
| 6 | 验收标准可测试 | 使用 Given-When-Then 格式；验收标准覆盖功能、交互、状态和边界，而非只写"功能正常" | 标注`[验收不可测试]`，建议重写为 Given-When-Then 格式 |
| 7 | 数据埋点完整 | 核心操作路径（进入、创建、提交、成功、失败）均有埋点；埋点属性含操作对象、来源、结果等关键维度 | 标注`[埋点缺失]`，列出核心路径缺少的埋点事件 |
| 8 | 无技术方案 | PRD 只描述"做什么"和"为什么"，不描述"怎么实现"；无数据库选型、编程语言、框架选型等技术实现细节 | 标注`[技术越界]`，删除技术实现描述，改为性能或安全要求 |
| 9 | 优先级明确 | 功能模块标注 P0/P1/P2；P0 = 必须上线，P1 = 强烈建议，P2 = 锦上添花 | 标注`[优先级未标]`，要求补充 P0/P1/P2 标注 |
| 10 | 排期有依据 | 工时评估考虑开发/测试/联调/上线四个阶段；排期不短于最短合理工时 | 标注`[排期不合理]`，列出缺失的阶段或过短的工时 |

## 自检执行规则

- **触发时机**：S7 正式产物写入完成后、S8 一致性审查之前。
- **执行方式**：逐项对照清单，标注每项是否通过。
- **不通过项处理**：每项给出具体标注和修改建议，不得只写"不通过"。
- **全部通过**：输出自检结果摘要，进入 S8。
- **部分不通过**：列出不通过项及修改建议，询问用户是否接受风险继续进入 S8，或返回修改。
- **自检结果必须写入对话**：自检结论和标注不作为单独文件，只出现在对话输出中。

## PRD 反模式防护

自检时同步检查以下反模式。命中时标注并提示修改建议：

| 反模式 | 表现 | 正确做法 |
|--------|------|----------|
| 需求镀金 | 一期写了 50 个功能点，未划分 MVP 范围 | 划分 MVP/V1.1/V2，一期聚焦核心 3-5 个功能 |
| 伪需求 | "用户可能需要..."没有数据或调研支撑 | 标注需求来源：用户反馈 / 数据分析 / 竞品参考 / 业务判断；无来源时标为 `[来源:假设-待验证]` |
| 交互越界 | PRD 里写了按钮颜色、字号、布局细节 | 只描述信息层级和交互逻辑，视觉交给设计师 |
| 技术越界 | PRD 指定用 Redis 缓存、MySQL 存储 | 只描述性能要求（如 "<2 秒"），实现交给技术 |
| 规则黑洞 | "按照业务规则处理"但不写具体规则 | 穷举每条规则的触发条件、处理逻辑和边界值 |

## 需求来源标注规则

每个功能模块和业务规则的来源必须标注。可选标签：

- `[来源:用户反馈]` — 有真实用户访谈、工单或反馈记录
- `[来源:数据分析]` — 有埋点、日志或业务数据支撑
- `[来源:竞品参考]` — 参考了竞品功能或行业实践
- `[来源:业务判断]` — 产品经理或业务方主观判断
- `[来源:假设-待验证]` — 无任何支撑，需在验收或迭代中验证

无来源标注的需求在自检时标注为 `[来源:假设-待验证]`。

## 与 S8 一致性审查的关系

本自检清单是 S8 的前置步骤。自检通过后进入 S8 审查跨产物一致性；自检不通过时，即使 S8 审查通过，产物仍需返回修改。

两者职责划分：
- **本清单**：检查单份 PRD / 模块详设文档自身的完整度。
- **S8 审查**：检查主 PRD 与模块详设之间、产物与 checkpoint / MVP 范围之间是否一致。
