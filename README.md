# PM Assistant

PM Assistant 是一个 Claude Code skill，用来把粗糙的产品想法推进成可执行的产品判断、MVP 范围、一份完整可读的 `PRD.md`，以及按模块拆分的真实颗粒度详设，必要时再补充开发交接文件。

它面向开发工程师和 AI 编程用户：你可能实现速度很快，但在动手前需要更强的产品判断、范围收口和验收定义。

## 它做什么

PM Assistant 以“产品教练 + 评审式拷问”的方式工作。它会挑战模糊想法，强制做 C 端 / B 端 / 混合型分流，收窄 MVP，并把确认过的产品决策整理成工程师能使用的产物。

适合用于：

- 在实现前澄清产品想法。
- 判断产品是 C 端、B 端还是混合型。
- 明确目标用户、核心场景、痛感、替代方案和证据缺口。
- 把过大的想法裁剪成可验证的 MVP。
- 生成一份整合背景、痛点、环境、选型假设、流程、模块总览、交互、原型布局、验收和风险的完整 PRD。
- 为复杂功能生成 `modules/` 模块详设，把入口、主客态、页面线框、状态机、跳转回流、文案、数据、埋点和验收写到真实产品迭代颗粒度。
- 在需要时额外生成开发交接。
- 把产品判断追问到页面结构、视觉交互原则、核心组件和按钮级交互细节。
- 给设计 / 原型工具准备足够清晰的页面线框、交互状态和视觉约束输入。

不适合用于：

- 编写生产级代码。
- 已冻结需求后的纯 UI 视觉微调。
- 泛商业战略报告。
- 项目管理进度追踪。
- 不需要产品判断的普通文案润色。

## 快速开始

用一个粗糙产品想法启动：

```text
使用 $pm-assistant 评审我的产品想法：
我想做一个帮助自由职业者管理客户需求的 App。
```

也可以直接要求具体产物：

```text
使用 $pm-assistant 把这个想法收口成 MVP 范围和 PRD。
```

```text
使用 $pm-assistant 生成 PRD 和模块详设，后续我会交给设计工具继续做视觉稿。
```

首次运行时，如果没有产品名，PM Assistant 会先要求确认产品名，创建轻量 checkpoint，然后从产品类型分流开始。

## 核心流程

PM Assistant 只使用一套阶段模型：`S0-S8`。

| 阶段 | 目的 |
|------|------|
| S0 | 确认产品名并初始化 checkpoint |
| S1 | 分流产品类型：C 端、B 端或混合型 |
| S1.5 | 识别产品环境、现有系统、技术依赖、合规和发布约束 |
| S2 | 澄清目标用户和核心场景 |
| S3 | 评估问题价值、替代方案和证据 |
| S4 | 定义 MVP 范围和非目标 |
| S5 | 将 MVP 转成规则、状态、流程和边界情况 |
| S5.5 | 拷问产品维度、视觉交互原则和核心按钮细节 |
| S5.6 | 对需要下沉的模块执行真实颗粒度详设拷问 |
| S6 | 定义验收标准和成功信号 |
| S7 | 生成版本化正式产物 |
| S8 | 执行一致性审查并准备下一轮迭代 |

S1-S6 以及按条件必跑的 S1.5 / S5.6 每个阶段闭合时都会输出：

- 已确认判断。
- 未确认阻塞项。
- 推荐答案。
- 下一步。

模糊回答不会被当作已确认决策。

## 记忆与产物

澄清阶段只维护轻量 checkpoint：

```text
pm-memory/
└── PROJECT-STATE.md
```

正式产物写入版本化目录：

```text
pm-output/
└── iterations/
    └── <version>/
        ├── PRD.md
        ├── modules/
        │   ├── 01-<模块名>.md
        │   └── 02-<模块名>.md
        ├── 开发交接.md
        ├── 迭代摘要.md
        ├── 决策日志.md
        ├── 未决问题.md
        └── 调研材料.md
```

PM Assistant 不会默认生成所有文件。默认正式输出是一份完整 `PRD.md`；如果版本包含多个模块、已有产品迭代、多入口、邀请 / 分享 / 授权、H5-App 跳转、积分 / 付费、审核或异步任务，会额外生成 `modules/` 模块详设。当用户要求开发实现交接时，再生成 `开发交接.md`。`PRD.md` 保持英文文件名，模块和交接文件使用中文文件名。

环境信息采用分层写法：主 `PRD.md` 写全局产品环境、用户环境、现有系统、技术依赖、合规安全和发布验证；每个 `modules/*.md` 写“模块环境继承与覆盖”，说明该模块继承了哪些全局约束，又补充或覆盖了哪些局部约束。

S1-S6 闭合、S1.5 已完成，且触发模块详设时 S5.6 也已完成后，PM Assistant 会先在对话中输出产品决策摘要，再询问是否保存为完整 `PRD.md`。用户拒绝保存时，只保留 `pm-memory/PROJECT-STATE.md` checkpoint。

写入正式产物前，PM Assistant 会检查 `pm-output/iterations/` 和 `pm-memory/OUTPUTS.md` 中已有版本，避免只凭 checkpoint 判断首版并覆盖旧产物。若目标版本已存在，默认创建新版本；只有用户明确要求覆盖或修正当前版本，才写入同名版本。

正式产物生成后，PM Assistant 必须进入 S8 一致性审查。只有 PRD 内的产品决策、流程、交互、原型布局、验收、风险之间没有冲突，且所有模块详设都能追溯到主 PRD 的模块目标和范围，并完成必要状态更新后，才能宣称本次正式输出完成。若生成了开发交接，也必须检查它与 PRD 和模块详设不冲突。

## 原型边界

默认情况下，PM Assistant 只在产品文档中定义原型要求：

- `PRD.md` 中的原型布局和交互章节
- `modules/` 中的页面 / 弹层 / 卡片线框图
- 交互需求
- 页面范围
- 主任务路径
- 关键状态
- 业务规则
- 产品维度判断
- 视觉交互原则
- 核心按钮细节
- 验收标准

具体视觉稿、HTML / CSS / SVG 原型、截图验证和 `design-output/` 不属于 PM Assistant 职责。后续由设计 / 原型工具基于 `PRD.md` 和 `modules/` 继续处理。

## 外部调研

当产品判断依赖市场、竞品、行业、用户或趋势事实时，PM Assistant 会在 S3 询问是否联网验证。

如果跳过联网或当前环境无法联网，相关判断会被标记为未验证假设，并进入风险或验证计划。

## 项目文件

- `SKILL.md`：主 skill 指令、阶段流程、门禁和行为规则。
- `agents/openai.yaml`：展示元数据和默认 prompt。
- `references/artifact-templates.md`：版本化产品产物模板。
- `references/environment-interrogation.md`：S1.5 环境识别与约束分流清单。
- `references/memory-rules.md`：checkpoint、进度索引和正式产物索引规则。
- `references/module-detail-interrogation.md`：S5.6 模块详设拷问清单。
- `references/formal-output-rules.md`：S7 正式输出、版本检查和完成门禁。
- `references/product-visual-interaction-interrogation.md`：产品维度、视觉交互原则和按钮级交互拷问清单。
- `issue-ledger.md`：Agent PM 审查问题台账。

## 设计原则

- 开发工程师是第一用户。
- 深度追问前必须先做 C / B / 混合型分流。
- 正式产物必须基于闭合的产品决策。
- 可以给推荐答案，但关键决策必须由用户确认。
- 弱需求应该被挑战，而不是被包装。
- 产品判断必须落到页面、状态、组件和按钮行为。
- 未验证证据可以继续推进，但必须记录为风险。
- Checkpoint 只用于恢复，不是完整文档库。
- 正式完成前必须检查 PRD、模块详设和开发交接之间的一致性。
