# 产品视觉交互拷问清单

当 PM Assistant 需要生成 PRD、模块详设、页面线框、视觉交互约束或开发交接时读取本文件。

目标不是替 UI 设计做审美决定，而是把产品判断压实到界面结构、视觉优先级、组件状态和按钮交互细节。任何影响页面骨架、信息架构、主 CTA、关键组件、风险提示或验收标准的判断，都不能靠 PM Assistant 擅自补齐。

## 一、产品维度拷问

### 1. 用户与角色

- 谁是每天真正操作的人？
- 谁只看结果但不操作？
- 谁审批、付费、承担风险或被结果影响？
- 不同角色看到的信息、按钮和权限是否不同？
- 如果只能优先服务一个角色，优先谁，为什么？

### 2. 场景与压力

- 用户是在高频、低频、紧急、碎片化还是长时间专注场景下使用？
- 使用设备、屏幕空间、网络环境、协作环境有什么约束？
- 用户进入页面时最想确认什么？
- 用户最怕犯什么错？
- 哪些信息必须首屏可见，哪些可以延后？

### 3. 任务与路径

- 主任务是什么，完成它最短路径有几步？
- 次任务是什么，是否应该弱化？
- 用户如何退出、取消、返回、撤销或重新开始？
- 失败后用户下一步是什么？
- 页面是否鼓励用户做正确的下一步，而不是只展示信息？

### 4. 价值与优先级

- 第一屏必须证明什么价值？
- 主 CTA 是否直接对应 MVP 的核心验证动作？
- 哪个指标或行为证明这个界面设计有效？
- 哪些功能看起来有吸引力但会稀释 MVP？
- 哪些内容是信任建立必需，哪些只是装饰？

### 5. 业务规则与风险

- 哪些业务规则必须在界面上显性表达？
- 哪些规则可以隐藏在系统行为里，但必须有反馈？
- 哪些操作不可逆、影响他人、影响账务、影响权限或影响数据完整性？
- 用户操作前是否知道影响范围？
- 系统失败时是否能保护用户当前进度？

## 二、产品到视觉转换

### C 端产品

优先拷问：
- 首次使用触发点是否清晰。
- 主 CTA 是否降低行动门槛。
- 情绪反馈是否服务留存，而不是只做热闹。
- 文案是否让用户知道自己会获得什么。
- 视觉是否过度承诺了产品能力。

常见禁止项：
- 为了好看隐藏主任务。
- 首屏只讲概念，不给行动入口。
- 用强营销视觉掩盖价值证据不足。

### B 端产品

优先拷问：
- 信息密度是否支持扫读和比较。
- 状态、权限、审批、异常是否足够透明。
- 批量操作是否有影响范围预览。
- 错误恢复和审计线索是否清楚。
- 页面是否支持重复、高频、低情绪波动的工作流。

常见禁止项：
- 用营销式大卡片稀释工作台效率。
- 把关键状态藏在二级页面。
- 危险操作缺少确认、撤销或影响范围说明。

### 混合型产品

优先拷问：
- C 端用户和 B 端客户的视觉优先级是否分开。
- 哪一侧是 MVP 启动侧。
- 两侧价值是否互相依赖。
- 是否有一侧界面服务另一侧的业务目标。
- 是否需要分别定义前台体验和后台管理体验。

常见禁止项：
- 用一套页面逻辑覆盖两类用户。
- C 端追求转化，B 端却看不到业务收益。
- B 端需要管控，C 端却被过度约束。

## 四、按钮级交互拷问

每个核心按钮必须问清以下内容：

| 维度 | 必须回答的问题 |
|------|----------------|
| 产品目的 | 这个按钮推动哪个产品目标或用户任务？ |
| 文案 | 用户看到的按钮文案是什么？是否需要动态文案？ |
| 位置 | 它在哪个页面、组件、状态下出现？ |
| 前置条件 | 需要哪些数据、权限、选择项或校验通过？ |
| 启用规则 | 什么时候可点击？什么时候禁用？ |
| 禁用解释 | 禁用时用户如何知道原因？ |
| 点击行为 | 点击后立即发生什么？同步还是异步？ |
| 加载反馈 | 请求中按钮、页面和相关区域怎么表现？ |
| 成功反馈 | 成功后停留、跳转、关闭、刷新还是进入下一步？ |
| 失败反馈 | 失败信息在哪里出现？用户怎么重试？ |
| 防重复提交 | 是否需要锁按钮、幂等 key、节流或确认？ |
| 权限不足 | 无权限时隐藏、禁用还是展示并解释？ |
| 风险提示 | 是否需要确认弹窗、影响范围预览、撤销入口？ |
| 验收方式 | 如何用测试或人工检查确认它正确？ |

### 危险操作按钮

额外必须确认：
- 是否不可逆。
- 是否影响他人、账务、权限或核心数据。
- 是否需要输入确认词。
- 是否需要展示影响对象数量和列表。
- 是否支持撤销、回滚或恢复。

### 表单提交按钮

额外必须确认：
- 校验在输入时、失焦时还是提交时触发。
- 错误定位到字段、区域还是全局提示。
- 是否支持保存草稿。
- 提交失败后是否保留用户输入。
- 离开页面是否需要未保存提醒。

### 批量操作按钮

额外必须确认：
- 选择范围是当前页、筛选结果还是全部数据。
- 是否存在部分对象无权限。
- 是否需要影响范围预览。
- 部分失败时如何展示成功项和失败项。
- 是否支持撤销或重新执行失败项。

## 五、页面线框与交互输入门禁

以下任一问题未闭合时，不得生成最终 PRD / 模块详设中的页面线框和交互规则，只能生成候选草案：

- 页面范围不清。
- 主 CTA 未确认。
- 首屏信息优先级未确认。
- 核心按钮缺少启用 / 禁用 / 反馈规则。
- 危险操作缺少风险提示。
- B 端权限、审批或批量操作未拆清。
- C 端转化动作和反馈方式未拆清。
- 混合型两侧用户的页面目标混在一起。
- 验收标准无法检查按钮、状态和主路径是否正确。

## 六、PRD / 模块详设必填项

写入 `PRD.md` 或 `modules/<序号>-<模块名>.md` 时，相关页面和交互章节必须包含：

- 产品名称和版本。
- 产品类型：C 端 / B 端 / 混合型。
- 目标用户与关键场景。
- 页面范围。
- 首屏优先级。
- 主任务路径。
- 主 CTA 与按钮细节。
- 关键状态和反馈策略。
- 业务规则和权限规则。
- 视觉交互原则。
- 禁止的视觉方向。
- 非目标。
- 验收标准。
